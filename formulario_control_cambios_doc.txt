1. INTRODUCCION Y ENFOQUE DIGITAL
================================================================================

1.1 PROPOSITO DEL FORMULARIO DENTRO DEL CICLO DE VIDA DEL SISTEMA
-----------------------------------------------------------------

El formulario de Control de Cambios constituye el nucleo del proceso de gestion
de modificaciones para el Sistema de Boletos y Gestion de Flota de Transporte,
una aplicacion web construida sobre el stack tecnologico Laravel + Livewire +
Tailwind CSS + MySQL. Este sistema opera como una cooperativa de transporte que
administra rutas, horarios, unidades (buses), venta de pasajes (tanto en
ventanilla fisica como mediante cliente web) y la infraestructura subyacente de
base de datos.

El formulario se integra en las siguientes fases del ciclo de vida del software:

  a) Fase de Planificacion (ITIL v4 — Service Value Chain: Plan):
     Permite capturar solicitudes de cambio provenientes de cualquier stakeholder
     del ecosistema (Administrador, Oficinista, Chofer, Pasajero o Developer),
     estructurandolas como artefactos trazables con identificador unico (formato
     SC-YYYYMMDD-XXX para solicitudes, RC-YYYYMMDD-XXX para reportes).

  b) Fase de Desarrollo (DevOps — Continuous Integration):
     El segundo tab ("Reporte del Cambio") vincula cada solicitud con su
     implementacion tecnica real: GitHub Issue ID, Feature Branch, Pull Request
     Link, Commit Hash y estado de validacion en la rama Sandbox (test-all-
     features). Esto materializa el principio de Infrastructure as Code y
     Change Advisory Board (CAB) digital.

  c) Fase de Validacion y Despliegue (ITIL v4 — Change Enablement):
     El pipeline de estados (Propuesto -> En Desarrollo -> Validado en Sandbox
     -> Aprobado en Code Review -> Mergado a Develop -> Desplegado en
     Produccion) refleja un flujo de aprobacion escalonado que previene la
     introduccion de codigo no verificado en ramas compartidas.

  d) Fase de Auditoria y Gobernanza (Software Auditing):
     La exportacion automatica a JSON genera un registro inmutable de cada
     cambio, incluyendo analisis de riesgos, plan de contingencia/rollback,
     aprobacion digital multi-rol e historial de versiones.


1.2 EL PORQUE: MIGRACION DE MODELO ESTATICO A FLUJO TRANSACCIONAL DIGITAL
-------------------------------------------------------------------------

La decision de migrar de un formulario estatico "tipo papel fisico" a un flujo
transaccional digital e interactivo se fundamenta en los siguientes principios
de ingenieria de software:

  a) Eliminacion de Friccion Cognitiva (Principio de Usabilidad de Nielsen):
     Un formulario estatico obliga al usuario a completar campos irrelevantes
     para su contexto. Por ejemplo, un Chofer que reporta un problema en la
     ventanilla no necesita conocer conceptos como "Migration de Base de Datos"
     o "Feature Branch". El formulario dinamico adapta su interfaz segun el rol
     del solicitante, reduciendo la carga cognitiva y la tasa de errores de
     entrada de datos.

  b) Trazabilidad Automatizada vs. Trazabilidad Manual:
     En un modelo de papel fisico, las firmas manuales se pierden, las fechas
     se escriben de forma inconsistente y no existe vinculacion automatica con
     el repositorio de codigo. El modelo digital genera identificadores unicos,
     timestamps automaticos y campos de trazabilidad Git/GitHub que crean un
     hilo de auditoria verificable criptograficamente (via commit hashes).

  c) Prevencion de Regresiones en Entornos Colaborativos:
     En un equipo de desarrollo donde multiples desarrolladores trabajan
     simultaneamente en la rama develop, la ausencia de un proceso formal de
     control de cambios provoca conflictos de_merge_, sobrescritura de
     funcionalidades y rotura de entornos locales de los companeros. El
     formulario digital impone una disciplina de Change Management que alinea
     las practicas del equipo con los estandares de ITIL v4 (practica de
     Change Enablement) y DevOps (GitFlow con validacion en Sandbox).

  d) Estandarizacion de la Informacion:
     Un formulario de papel permite respuestas ambiguas, incompletas o
     inconsistentes. El formulario digital aplica validacion en tiempo real
     (campos obligatorios, seleccion obligatoria de rol, tipo de cambio y
     prioridad), garantizando que cada registro contenga la informacion minima
     necesaria para su procesamiento downstream.

  e) Integracion Nativa con el Ecosistema DevOps:
     El formulario fue disenado desde su concepcion para exportar datos en
     formato JSON estructurado, listo para ser consumido por la GitHub API
     (creacion automatica de Issues), pipelines de CI/CD, o sistemas de
     monitoreo. El codigo incluye una implementacion comentada y lista para
     activar de la integracion con la GitHub REST API v3, lo que demuestra
     una arquitectura preparada para evolucion (Open-Closed Principle).

2.2 EL PORQUE: POR QUE LOS ROLES UNIVERSITARIOS ERAN INCORRECTOS
----------------------------------------------------------------

En versiones previas del formulario, se utilizaron roles genericos de contexto
academico como "Estudiante" y "Docente". Esta decision constituyo un error de
modelado de dominio por las siguientes razones:

  a) Violacion del Principio de Ubiquitous Language (Domain-Driven Design):
     El lenguaje del software debe reflejar el lenguaje del dominio de negocio.
     En una cooperativa de transporte, los actores reales son Administradores,
     Oficinistas, Choferes y Pasajeros. Introducir terminos como "Estudiante"
     crea una disonancia semantica que confunde a los usuarios reales y
     dificulta la comunicacion entre stakeholders tecnicos y no tecnicos.

  b) Falta de Correspondencia con los Casos de Uso Reales:
     Un "Estudiante" no tiene ningun caso de uso en un sistema de venta de
     pasajes. Un "Pasajero" si: consulta rutas, compra boletos, cancela
     viajes. Un "Docente" no tiene ningun caso de uso. Un "Administrador" si:
     gestiona rutas, horarios y flota. El mapeo incorrecto de roles impide
     que el formulario capture informacion relevante para el dominio.

  c) Implicaciones de Seguridad y Control de Acceso (RBAC):
     En un sistema de produccion real, los roles definen permisos de acceso
     (Role-Based Access Control). Un Chofer no debe tener acceso al panel
     administrativo. Un Pasajero no debe poder modificar rutas. Si el
     formulario utiliza roles ficticios, no puede integrarse correctamente con
     el sistema de autorizacion del backend (Laravel Gates/Policies), lo que
     genera vulnerabilidades potenciales de escalada de privilegios.

  d) Credibilidad ante Evaluacion Academica y Profesional:
     Un proyecto universitario que simula un sistema de transporte debe
     demostrar comprension del dominio de negocio. Utilizar roles reales
     demuestra madurez en el analisis de requisitos y alineacion con las
     practicas de la ingenieria de software profesional.

  e) Separacion de Responsabilidades (Separation of Concerns):
     Cada rol real tiene responsabilidades y perspectivas distintas sobre el
     sistema. Un Chofer reporta problemas operativos; un Developer implementa
     soluciones tecnicas. El formulario dinamico reconoce esta separacion y
     adapta la interfaz en consecuencia, algo que seria imposible con roles
     genericos que no reflejan la realidad del dominio.

3. LOGICA REACTIVA DINAMICA (TIPO DE CAMBIO)
================================================================================

3.1 DETALLE DE LAS OPCIONES MOSTRADAS SEGUN PERFIL DE USUARIO
-------------------------------------------------------------

El formulario implementa un mecanismo de renderizado condicional que muestra
diferentes opciones de "Tipo de Cambio" segun el rol seleccionado. La logica
se define en las constantes JavaScript:

  ROLES_USUARIO  = ['Pasajero', 'Chofer']
  ROLES_TECNICO  = ['Developer']
  ROLES_HIBRIDO  = ['Administrador (Operativa)', 'Oficinista (Ventanilla)']

  a) Para usuarios NO TECNICOS (Pasajero, Chofer):
     Se muestra el grupo #sol-tipo-usuario con tres opciones en lenguaje
     natural y orientado al negocio:

       - "Reportar un Problema"
         Descripcion: "Algo no funciona correctamente en el sistema"

       - "Sugerir una Nueva Idea / Funcion"
         Descripcion: "Propuesta de mejora o funcionalidad nueva"

       - "Mejora Visual o de Comodidad"
         Descripcion: "Cambios de diseno, apariencia o experiencia"

  b) Para usuarios TECNICOS (Developer):
     Se muestra el grupo #sol-tipo-tecnico con cinco opciones en terminologia
     de ingenieria de software:

       - "Nueva Regla de Negocio (Backend)"
         Descripcion: "Backend — Logica de negocio, validaciones"

       - "Ajuste de Interfaz / UX Avanzado"
         Descripcion: "UI/UX avanzado — Rediseño, flujos"

       - "Correccion de Error (Bugfix)"
         Descripcion: "Bugfix — Fallo en funcionalidad"

       - "Modificacion de Base de Datos (Migration)"
         Descripcion: "Migration — Cambio de esquema MySQL"

       - "Optimizacion de Codigo / Rendimiento"
         Descripcion: "Cache, queries N+1, rendimiento"

  c) Para roles HIBRIDOS (Administrador, Oficinista):
     Se muestra el grupo tecnico (#sol-tipo-tecnico) pero con un mensaje
     contextualizado: "Como [Rol], seleccione la categoria tecnica del cambio
     propuesto." Esto reconoce que estos roles tienen conocimiento parcial del
     sistema y pueden identificar categorias tecnicas sin ser desarrolladores.


3.2 EL PORQUE: SEPARACION DE ABSTRACCIONES COMO PRINCIPIO DE UX
---------------------------------------------------------------

Forzar a un Chofer u Oficinista a seleccionar terminos como "Modificacion de
BD", "Migration" o "Queries N+1" constituye una pesima practica de User
Experience (UX) y de diseno de interfaces por las siguientes razones tecnicas:

  a) Violacion del Principio de Abstraccion Adecuada:
     En ingenieria de software, cada capa del sistema debe exponer solo la
     abstraccion necesaria para su audiencia. Un Chofer interactua con el
     sistema a nivel de negocio (reportar que "el boton de venta no funciona"),
     no a nivel tecnico ("hay un error en la migracion de la tabla boletos").
     Exponer abstracciones tecnicas a usuarios no tecnicos genera confusion,
     errores de clasificacion y frustracion.

  b) Ley de Postel (Principio de Robustez):
     "Se conservador en lo que envias, se liberal en lo que aceptas." El
     formulario acepta solicitudes de cualquier stakeholder (liberal en la
     entrada) pero las traduce internamente a categorias tecnicas procesables
     (conservador en la salida). Esta traduccion de abstracciones es
     fundamental para la interoperabilidad entre equipos tecnicos y no tecnicos.

  c) Reduccion de la Tasa de Error en la Clasificacion:
     Cuando un usuario no tecnico enfrenta terminologia que no comprende, tiende
     a seleccionar opciones al azar o abandonar el formulario. Al presentar
     opciones en su propio lenguaje ("Reportar un Problema" vs. "Correccion de
     Error (Bugfix)"), se incrementa significativamente la precision de la
     clasificacion, lo que a su vez mejora la asignacion automatica de
     prioridades y la estimacion de esfuerzo del equipo de desarrollo.

  d) Alineacion con el Patron de Diseno "Facade":
     La interfaz de usuario actua como una fachada (Facade Pattern) que oculta
     la complejidad del sistema tecnico al usuario final. El Chofer ve
     "Reportar un Problema"; el sistema internamente puede mapear esto a un
     Issue de tipo "bug" en GitHub con labels apropiados. Esta capa de
     abstraccion es esencial para la escalabilidad del sistema.

  e) Implementacion Tecnica (Event-Driven Reactivity):
     La reactividad se logra mediante un event listener attachado a cada
     .card-option del grupo #sol-rol. Al hacer click, se ejecuta la funcion
     updateTipoDeCambio() con un setTimeout de 50ms para asegurar que el estado
     de seleccion se haya actualizado antes de evaluar la logica condicional.
     Esta aproximacion es mas eficiente que un framework reactivo completo
     porque solo se ejecuta cuando el usuario interactua especificamente con el
     selector de rol, no en cada renderizado del DOM.

4. OPTIMIZACION DE RENDIMIENTO (VANILLA JS VS FRAMEWORKS)
================================================================================

4.1 EXPLICACION DE LA IMPLEMENTACION
-------------------------------------

El formulario esta implementado integramente en JavaScript Nativo (Vanilla JS),
sin dependencias externas de ningun tipo. No se utilizan CDNs, ni frameworks
reactivos (Alpine.js, React, Vue), ni librerias de manipulacion del DOM
(jQuery). La implementacion se basa en:

  a) API nativa de seleccion de elementos:
     - document.querySelectorAll() para seleccion multiple
     - document.getElementById() para seleccion por identificador
     - element.closest() para navegacion ascendente del DOM

  b) API nativa de eventos:
     - addEventListener('click', callback) para interacciones de usuario
     - addEventListener('change', callback) para inputs de archivo
     - addEventListener('submit', callback) para envio de formularios
     - e.preventDefault() para control del flujo de envio

  c) Manipulacion directa del DOM:
     - classList.add() / classList.remove() / classList.toggle() para estados
     - style.display = 'block' / 'none' para visibilidad condicional
     - style.animation para transiciones dinamicas

  d) Generacion de datos y exportacion:
     - Blob API para creacion de archivos JSON en memoria
     - URL.createObjectURL() / URL.revokeObjectURL() para descarga
     - JSON.stringify() con indentacion para legibilidad


4.2 EL PORQUE: SUSTENTO DE INGENIERIA PARA NO USAR DEPENDENCIAS EXTERNAS
------------------------------------------------------------------------

La decision de no utilizar dependencias externas (como un CDN de Alpine.js) se
sustenta en los siguientes principios de ingenieria de software:

  a) Principio KISS (Keep It Simple, Stupid):
     El formulario tiene una complejidad funcional moderada: tabs, seleccion de
     tarjetas, validacion basica, exportacion JSON y un modal de confirmacion.
     Estas funcionalidades se implementan en aproximadamente 400 lineas de
     JavaScript nativo. Introducir un framework reactivo para esta complejidad
     constituye overengineering (sobreingenieria): se anade complejidad
     arquitectonica sin beneficio proporcional.

  b) Reduccion de la Carga de Red y Tiempo de Carga:
     Un CDN de Alpine.js (version minima) anadiria aproximadamente 15KB
     comprimidos (gzip) al payload inicial de la pagina. Para un formulario que
     puede ejecutarse en entornos con conectividad limitada (oficinas de
     terminales de autobus, zonas rurales), cada kilobyte cuenta. Al eliminar
     la dependencia, se reduce el Time to Interactive (TTI) y se mejora el
     rendimiento percibido por el usuario.

  c) Eliminacion de Puntos de Fallo Externos:
     Una dependencia de CDN introduce un punto de fallo externo: si el CDN
     experimenta una interrupcion, el formulario deja de funcionar. En un
     sistema de gestion de transporte donde el formulario puede ser critico
     para reportar incidencias operativas, la disponibilidad es un requisito
     no negociable. Vanilla JS garantiza que el formulario funciona
     exclusivamente con recursos locales.

  d) Principio de Minima Dependencia (YAGNI — You Ain't Gonna Need It):
     Los frameworks reactivos brillan en aplicaciones con estado complejo,
     renderizado condicional frecuente y actualizaciones parciales del DOM.
     Este formulario tiene un flujo lineal: el usuario completa campos, hace
     click en "Enviar", y se descarga un JSON. No hay actualizaciones en
     tiempo real, no hay sincronizacion con servidor, no hay estado compartido
     entre componentes. Las caracteristicas reactivas de un framework son
     innecesarias para este caso de uso.

  e) Mantenibilidad y Curva de Aprendizaje:
     Cualquier desarrollador que conozca JavaScript basico puede leer, entender
     y modificar el codigo del formulario. Un framework como Alpine.js requiere
     conocimiento de su sintaxis especifica (x-data, x-bind, x-on, x-model),
     lo que crea una barrera de entrada para desarrolladores nuevos en el
     proyecto. En un contexto universitario donde los miembros del equipo
     rotan, la simplicidad del codigo es un activo valioso.

  f) Compatibilidad y Longevidad:
     JavaScript nativo es estable y retrocompatible. Las APIs utilizadas
     (querySelector, addEventListener, classList, Blob) estan soportadas en
     todos los navegadores modernos desde hace mas de una decada. Un framework
     externo puede quedar obsoleto, dejar de recibir mantenimiento o introducir
     breaking changes en sus versiones mayores, lo que obligaria a reescribir
     partes del formulario.

  g) Analisis Cuantitativo del Peso Total:
     El archivo HTML completo (CSS + HTML + JS) pesa aproximadamente 65KB.
     Con Alpine.js, pesaria ~80KB. La diferencia del 23% en peso no se
     justifica por las funcionalidades adicionales que aportaria el framework,
     dado que ninguna de ellas es requerida por los requisitos funcionales del
     formulario.

5. TRAZABILIDAD TECNICA Y FLUJO DE DEVOPS (REPORTE DEL CAMBIO)
================================================================================

5.1 DETALLE DE LOS CAMPOS DIGITALES INTEGRADOS
----------------------------------------------

El segundo tab del formulario ("Reporte del Cambio") implementa un sistema
completo de trazabilidad tecnica que reemplaza las firmas fisicas manuales de
los procesos de Change Management tradicionales. Los campos integrados son:

  a) GitHub Issue ID (formato: #123):
     Vincula el reporte con el Issue correspondiente en el repositorio de
     GitHub. Esto permite navegar bidireccionalmente entre el formulario y la
     discusion tecnica del cambio, incluyendo comentarios, asignaciones y
     labels.

  b) Feature Branch Name (formato: feature/SC-YYYYMMDD-XXX-descripcion):
     Documenta la rama de trabajo donde se implemento el cambio. La convencion
     de nombres incluye el identificador de la solicitud vinculada, lo que
     garantiza que cada rama sea trazable a su origen. Esta practica es
     fundamental en GitFlow para evitar ramas huerfanas o sin contexto.

  c) Pull Request (PR) Link (URL completa):
     Proporciona acceso directo al Pull Request donde se solicito la
     integracion del cambio a la rama develop. El PR contiene el diff completo,
     los comentarios de revision, el estado de los checks de CI y la decision
     de merge.

  d) Commit Hash (formato: abc1234):
     Identifica de forma unica e inmutable el commit que implementa el cambio.
     El hash SHA-1 de Git es una garantia criptografica de integridad: cualquier
     modificacion posterior al commit cambiaria el hash, lo que permite detectar
     alteraciones no autorizadas.

  e) Sandbox Testing Status (Pendiente / En ejecucion / Aprobado / Fallido):
     Documenta el resultado de la ejecucion de pruebas en el entorno Sandbox
     (rama test-all-features). Este campo es critico para prevenir que cambios
     no validados lleguen a la rama develop y rompan los entornos locales de
     los companeros de equipo.

  f) Technical Reviewer / Code Owner:
     Identifica al responsable de la revision de codigo. En GitHub, los Code
     Owners son desarrolladores con expertise en modulos especificos que deben
     aprobar los cambios antes de que se integren.

  g) Code Owner Approval (Aprobado / Requiere cambios / Rechazado):
     Formaliza la decision del Code Owner, reemplazando la firma manual en un
     documento fisico. Esta aprobacion digital es verificable, timestamped y
     vinculada al PR correspondiente.

  h) Validacion en Sandbox (test-all-features) — Grid de Pruebas:
     Seis modulos del sistema se evaluan individualmente con estado pass/fail:

       1. Operativa — CRUD rutas, horarios, buses
       2. Ventanilla — Venta manual, asientos
       3. Web Client — Catalogo, carrito, pagos
       4. Base de Datos — Migraciones, integridad
       5. Livewire Components — Reactividad
       6. Tailwind CSS — Responsive, visual

     Cada modulo tiene botones de estado (✅ / ❌) que permiten documentar
     exactamente que partes del sistema fueron afectadas y validadas.

  i) Analisis de Impacto (Alcance, Riesgo, Backward Compatibility,
     Dependencias):
     Evalua proactivamente las consecuencias del cambio antes de su
     implementacion, alineandose con la practica de ITIL v4 "Change Enablement"
     que requiere una evaluacion de riesgo formal antes de aprobar cualquier
     cambio.

  j) Analisis de Riesgos (tabla de 3 riesgos con Probabilidad, Impacto y
     Mitigacion):
     Cada riesgo se evalua cualitativamente (Baja/Media/Alta para
     probabilidad; Bajo/Medio/Alto para impacto) y se documenta una medida de
     mitigacion especifica.

  k) Plan de Contingencia / Rollback (4 pasos):
     Documenta el procedimiento para revertir el cambio si causa problemas en
     produccion, incluyendo comandos especificos:

       Paso 1: Descripcion del rollback
       Paso 2: git revert <commit-hash>
       Paso 3: php artisan migrate:rollback --step=1
       Paso 4: Verificacion post-rollback

  l) Aprobacion Digital (tabla multi-rol):
     Tres roles firman digitalmente el cambio:

       - Developer: Completado
       - Technical Reviewer: Aprobado / Rechazado
       - Product Owner / Admin: Aprobado / Rechazado

     Cada firma incluye nombre, estado y fecha, reemplazando las firmas
     manuscritas de los procesos tradicionales de Change Advisory Board.

  m) Historial de Versiones del Cambio:
     Registra la evolucion del cambio a traves de versiones (v1.0, v1.1, v1.2),
     incluyendo autor, fecha y descripcion de cada modificacion.


5.2 EL PORQUE: REEMPLAZO DE FIRMAS FISICAS POR AUDITORIA DIGITAL
----------------------------------------------------------------

La transicion de firmas fisicas manuales a un sistema de auditoria digital
basado en Git/GitHub se justifica por los siguientes argumentos:

  a) Verificabilidad Criptografica:
     Una firma manuscrita en un documento de papel puede falsificarse. Un
     commit hash de Git es un digest SHA-1 que verifica la integridad de todo
     el arbol de archivos. Si alguien modifica un archivo despues del commit,
     el hash cambia y la alteracion es detectable. Esto proporciona un nivel
     de seguridad muy superior al de las firmas fisicas.

  b) Prevencion de Regresiones en Entornos Colaborativos:
     En un equipo donde multiples desarrolladores trabajan en la rama develop
     simultaneamente, un cambio no validado puede romper funcionalidades
     existentes. El proceso de validacion en Sandbox (test-all-features)
     actua como un gate de calidad: ningun cambio se integra a develop sin
     haber pasado las pruebas en los seis modulos del sistema. Esto previene
     el escenario donde "el codigo de un companero rompe mi entorno local",
     un problema comun en equipos que no implementan Change Management formal.

  c) Trazabilidad Bidireccional Completa:
     Con las firmas fisicas, es dificil rastrear que cambio causo un bug en
     produccion. Con el sistema digital, la cadena es completa:

       Solicitud (SC-YYYYMMDD-XXX)
         -> Reporte (RC-YYYYMMDD-XXX)
           -> GitHub Issue (#123)
             -> Feature Branch (feature/SC-...)
               -> Pull Request (PR #456)
                 -> Commit Hash (abc1234)
                   -> Merge a Develop
                     -> Despliegue a Produccion

     Cualquier punto de esta cadena es accesible desde cualquier otro punto,
     lo que facilita enormemente el debugging y la auditoria.

  d) Cumplimiento de Estandares de Auditoria de Software:
     Los estandares de auditoria de software (como ISO/IEC 27001 para
     seguridad de la informacion o SOX para control financiero) requieren
     evidencia documentada de que cada cambio fue autorizado, probado y
     aprobado. El formulario digital genera esta evidencia de forma
     automatica y estructurada (JSON), mientras que las firmas fisicas
     requieren escaneo, almacenamiento manual y busqueda manual.

  e) Eficiencia Operativa:
     Obtener tres firmas fisicas en un documento de papel requiere coordinar
     la presencia fisica de tres personas. La aprobacion digital permite que
     cada rol firme de forma asincrona, desde cualquier ubicacion, con
     timestamps automaticos que registran exactamente cuando ocurrio cada
     aprobacion.

  f) Inmutabilidad del Registro:
     Una vez que el JSON se exporta y se descarga, el registro es inmutable.
     No se puede modificar retroactivamente sin dejar evidencia (el hash del
     archivo cambiaria). En contraste, un documento de papel puede ser
     alterado, perdido o destruido sin dejar rastro.

7.2 GENERACION DE IDENTIFICADORES UNICOS
-----------------------------------------

  - Formato: {PREFIX}-YYYYMMDD-XXX
  - PREFIX: "SC" para Solicitudes de Cambio, "RC" para Reportes de Cambio
  - YYYYMMDD: Fecha actual en formato compacto
  - XXX: Numero aleatorio de 3 digitos (100-999)

  Ejemplo: SC-20260527-456

  Este formato garantiza unicidad practica para el volumen de cambios esperado
  en un proyecto universitario y proporciona informacion temporal inmediata.

7.3 EXPORTACION JSON
---------------------

Al enviar exitosamente, el formulario:

  a) Recopila todos los datos del formulario en un objeto JavaScript
     estructurado.
  b) Serializa el objeto a JSON con indentacion de 2 espacios (legibilidad).
  c) Crea un Blob con tipo MIME application/json.
  d) Genera una URL temporal con URL.createObjectURL().
  e) Dispara la descarga automatica del archivo.
  f) Revoca la URL temporal para liberar memoria (URL.revokeObjectURL()).

  Nombres de archivo:
    - Solicitudes: solicitud-SC-YYYYMMDD-XXX.json
    - Reportes: reporte-RC-YYYYMMDD-XXX.json

  La estructura JSON incluye todos los campos del formulario, organizados
  logicamente en sub-objetos (solicitante, detalles, trazabilidad, etc.),
  lo que facilita su consumo por sistemas downstream.

8. SEGURIDAD Y BUENAS PRACTICAS
================================================================================

8.1 MEDIDAS DE SEGURIDAD IMPLEMENTADAS
---------------------------------------

  a) Atributo novalidate en formularios:
     Permite validacion personalizada en JavaScript en lugar de la validacion
     nativa del navegador, proporcionando un feedback de error consistente y
     estilizado.

  b) Sanitizacion implicita:
     Al exportar a JSON, los datos se serializan mediante JSON.stringify(),
     que escapa automaticamente caracteres especiales, previniendo inyecciones
     en el archivo de salida.

  c) No se almacenan datos en localStorage o sessionStorage:
     Los datos existen solo en memoria durante la sesion del formulario y se
     exportan directamente al archivo JSON. No persisten en el navegador,
     reduciendo la superficie de ataque.

  d) Revocacion de Object URLs:
     Despues de la descarga, se ejecuta URL.revokeObjectURL() para liberar
     la referencia al Blob, previniendo memory leaks.

  e) GitHub API Token (integracion futura):
     El codigo de integracion con GitHub API incluye un placeholder para el
     token (ghp_XXXXXXXXXXXXXXXXXXXX). En produccion, este token debe
     almacenarse en variables de entorno del servidor, nunca en codigo del
     lado del cliente.


8.2 BUENAS PRACTICAS DE INGENIERIA
-----------------------------------

  a) Separacion de responsabilidades:
     El formulario separa claramente la logica de UI (tabs, cards, modals) de

  b) Funciones reutilizables:

     extension: se puede anadir la integracion sin cambiar la logica existente.

  e) Accesibilidad:
     - Labels asociados a sus inputs correspondientes
     - Atributos placeholder descriptivos
     - Estados visuales claros (selected, has-error, active)
     - Feedback de carga (spinner en boton de envio)
